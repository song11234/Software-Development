## 实际步骤

1. 通过 web_search 搜索关键词，定位官方及可信公开页面
2. 尝试直接 web_fetch 访问目标页，发现 SPA 页面 JS 渲染后无正文
3. 改用公开 API 接口获取结构化数据，遇风控 412 后换源
4. 通过文章 API 成功获取专栏正文和摘要
5. 从摘要中抽取兑换码候选，码中含日期线索时推导有效期限
6. 用 urllib 抓取第三方页面 HTML，正则抽取疑似兑换码
7. 发现 Python `\b` 在中英文交界失效，改为手工提取
8. 组装为符合 11 字段约定的 JSON，区分来源可信度
9. 对照字段约定逐条自检，修正冗余字段和错误标注
10. 最终产出结构化结果

## 中途失败或返工

| 问题 | 原因 | 修正方式 |
|:--|:--|:--|
| B站动态页面返回空白 | SPA，JS 动态渲染 | 改用 B站文章 API |
| B站动态 API 返回 412 | 风控拦截 | 标记"不适合自动抓取"，换用专栏页 |
| 脚本返回 0 条 | SPA 页面无静态文本 | 从 API 或 HTML 页面中手工提取 |
| 正则 `\b` 中英文边界失效 | Python 3 CJK 字符影响 | 手工提取，记录为已知限制 |
| JSON 多了冗余字段 | 未严格对照约定表 | 删除冗余，修正 status 值 |
| 来源与条目混淆 | 批量操作时未逐条核对 | 逐条核对 source 字段 |

## 可复用经验

**应该固定下来的步骤：**
- 抓取前先搜索定位公开页面 URL，确认页面类型（SPA/静态/API）
- SPA 页面优先找公开 API，API 不可用时标记为"不适合自动抓取"
- 第三方来源永远标 confidence=low、status=unverified
- 码中含日期线索时用于推导 valid_until
- 输出后逐条对照字段约定表自检

**需要视情况判断的内容：**
- 不同网站的风控策略不同，没有统一绕过方式
- 兑换码格式不统一，正则需根据实际调整
- 时间解析依赖原文表达方式，无法完全自动化
