## 质量分析

### 无 Skill 组问题

1. **无来源审计**：直接使用 genshin-builds.com、game8.co 等第三方网站，不区分官方/第三方，不标注可信度
2. **时间字段缺失**：完全无 valid_from/valid_until，用户无法判断码是否过期
3. **混入过期码**：game8 的直播码已标注 6月22日过期，仍被列入
4. **无去重**：多个来源的重复码未合并
5. **输出不可复用**：仅有表格，无 JSON，无法被程序消费
6. **无防诈骗提醒**：第三方来源的码有误导风险

### 有 Skill 组优势

1. **来源分层**：明确区分 official_social（B站专栏）和 third_party（其他），可信度分级
2. **时间规范化**：ISO 8601 格式，缺失信息标注 unknown 而非编造
3. **去重合并**：跨来源对比，GENSHINGIFT 和 BILIGIFT2025 识别为重复并保留优先来源
4. **errors 记录**：4 条不可抓取来源记录在案，便于后续人工补全
5. **结构化输出**：JSON 可直接被下游系统消费
6. **防诈骗提醒**：明确第三方码需核验

## 效率分析

| 维度 | 无 Skill | 有 Skill |
|:--|:--|:--|
| **一次性完成率** | 低（需追问来源、时间） | 高（11 字段一次性全覆盖） |
| **后续人工补全成本** | 高（需手动查时间、分来源） | 低（结构化数据可直接使用） |
| **可复现性** | 差（无固定流程，每次结果差异大） | 好（每次按 Skill 流程执行，输出格式一致） |
| **返工概率** | 高（容易遗漏关键字段） | 低（自查清单防止遗漏） |

## Token 消耗分析

| 项目 | 无 Skill | 有 Skill | 增量原因 |
|:--|:--|:--|:--|
| 用户提示词 | ~30 tokens | ~150 tokens（含 SKILL 提示词） | Skill 内容带来的输入 Token 增大 |
| 对话输出 | ~500 tokens | ~2000 tokens（表格+JSON+自查） | 输出内容丰富 4 倍 |
| 返工 Token | ~200 tokens（补搜第三方） | 0 | 有 Skill 无需返工 |
| **合计** | ~800 tokens | ~2200 tokens | 增量 ~1400 tokens |

> 虽然有 Skill 组单次 Token 消耗更大（约 2.75 倍），但信息密度和可用性远超无 Skill 组。且返工 Token 为零，减少了无效轮次的浪费。

## 结论

### 无 Skill 有效输出 ≈ 2/10
- 仅 10 个码的粗表格，缺关键字段
- 无法判断来源可靠性
- 无法判断时效性
- 无法被程序直接使用

### 有 Skill 有效输出 ≈ 9/10
- 14 条结构化记录，11 字段完备
- 来源分层清晰（official_social vs third_party）
- 时间字段归一化（ISO 8601 + unknown）
- JSON 可直接被程序消费
- 含 errors 记录 + 防诈骗提醒

### Skill 的核心价值体现

| 价值点 | 体现 |
|:--|:--|
| **减少重复说明** | 提示词中无需重复说明抓取边界和字段要求，Skill 已包含 |
| **减少返工** | 自查清单在输出前自查，无遗漏无错误，零返工 |
| **提升可复现性** | 每次执行遵循相同工作流，输出格式一致 |
| **接近标准** | 11 字段统一，ISO 8601 时间，unknown 标注一致 |
| **放大提示质量** | 短提示词 + Skill 加载 → 等价于极长提示词的产出质量 |

### Token 代价是否合理？

以本次实验为例，有 Skill 多消耗约 1400 tokens 但换来了：
- 4 倍的内容信息量
- 0 次返工（vs 无 Skill 的 1-2 次）
- 结构化 JSON 输出（vs 无 Skill 的纯表格）
- 来源分层 + 时效标注 + 可信度

**结论：多消耗的 ~1400 tokens 换来了质变级的输出提升，代价完全合理。**
